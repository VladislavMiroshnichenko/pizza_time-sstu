# Экзаменационный проект по ООП — Pizza Time

Консольное приложение на языке Python, моделирующее бизнес-логику работы пиццерии. Проект выполнен в качестве практического задания к экзамену по дисциплине **«Объектно-ориентированное программирование»** в СГТУ имени Гагарина Ю.А. (ИнПИТ).

## 📝 Описание проекта

Приложение симулирует полный цикл обработки заказа в заведении: от намерения клиента съесть пиццу до её приготовления поваром и автоматического оповещения через электронное табло. 

Взаимодействие построено на классических принципах объектно-ориентированного программирования (ООП) с разделением ответственности между независимыми компонентами приложения.

## 🏗️ Архитектура и основные сущности

Программа состоит из следующих ключевых классов:
1. `User` (Пользователь/Клиент) — обладает именем, умеет инициировать заказ в пиццерии, принимать уведомления от табло готовности и забирать свой заказ.
2. `Pizzeria` (Пиццерия) — центральный узел системы. Принимает заказы, регистрирует их в пуле, координирует процесс готовки и отправляет сигналы на табло.
3. `Pizza` (Пицца) — сущность, инкапсулирующая данные о конкретном кулинарном изделии (например, название).
4. `Order` (Заказ) — связующее звено, которое хранит в себе уникальный ID, ссылку на объект клиента (`User`), объект заказанной пиццы (`Pizza`) и текущий статус готовности.
5. `UI` (Интерфейс) — утилитарный класс со статическими методами для красивого форматирования вывода логов в консоль, реализующий принцип разделения ответственности (SRP).

## 🚀 Применение принципов ООП в Python

В ходе реализации проекта были применены и продемонстрированы следующие концепции:
* **Инкапсуляция и сокрытие данных:** Все критически важные внутренние поля классов (например, пул заказов `__orders_pool` или имя клиента `__name__`) скрыты с помощью механизма *Name Mangling* (двойное подчеркивание). Доступ к чтению организован безопасно через декораторы `@property`.
* **Полиморфизм и слабая связанность:** Пиццерия взаимодействует с абстрактным объектом заказа, вызывая методы уведомления у связанного с ним пользователя, не вдаваясь во внутреннюю структуру класса клиента.
* **Принцип DRY (Don't Repeat Yourself):** Логика отрисовки интерфейсных разделителей и заголовков вынесена в отдельный класс `UI`, что исключает дублирование кода `print()` в теле основной программы.
